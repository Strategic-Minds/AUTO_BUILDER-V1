# Passive Reverse Engineering System

## Purpose
Create a governed, mostly read-only system that can keep absorbing structure while the operator is offline.

The goal is maximum grounded coverage, not invented certainty. This system can move toward very high coverage of public and connected-source structure, but it cannot guarantee private internal details that are not available through lawful public evidence or approved connected sources.

## Operating rule
Run read-only by default while the operator is sleeping.

Escalate instead of acting when a task would require:
- credentialed live mutation
- bypassing access controls
- external messages or publishing
- destructive writes
- legal or compliance-sensitive actions

## Passive lanes
1. Public surface capture
Capture public pages, docs, pricing, onboarding surfaces, social proof, and visible UX flows.

2. Workflow trace extraction
Extract visible workflows from demos, help docs, forms, onboarding flows, and public process clues.

3. Capability diffing
Compare target systems against benchmark systems and record what is copyable, counterable, or should be ignored.

4. Asset and prompt extraction
Capture visible offer structure, hooks, content patterns, case-study shapes, reusable prompts, and template candidates.

5. Passive capability tests
Check internal route readiness, connector fallback state, browser-test health, queue readiness, and blocked surfaces.

## Suggested passive schedule
- Hourly: capability readiness checks and workaround queue generation
- Nightly: public surface capture, workflow trace extraction, asset and prompt extraction
- Daily: competitor or benchmark capability diffing

## Suggested outputs
- evidence ledger
- architecture map
- offer map
- queue and workflow map
- template extraction list
- capability delta report
- workaround queue
- build-card candidates for the best systems to replicate or counter

## GPT + Codex + Browser model
- GPT: planning, ranking, summarizing, hypothesis routing
- Codex: code and structure analysis, repo-facing patch planning, system decomposition
- Browser: public-surface capture, screenshots, visible flow tracing, passive validation

## Safety rule
Reverse engineer to the highest grounded level possible, but never claim hidden certainty. Always separate:
- Verified
- Inferred
- Could not verify
