## 8. Visual parity gate

Required visual parity:

`99.00% minimum`

Compare the deployed preview with the approved visual source across:

- composition and section order
- grids, spacing, alignment, and container widths
- typography, scale, weight, and line height
- colors, backgrounds, borders, shadows, and radii
- icons, imagery, and cropping
- navigation, headers, footers, cards, controls, and forms
- responsive transformations and mobile stacking
- hover, focus, selected, empty, loading, and error states

Minimum viewport coverage:

- desktop
- tablet
- mobile

Use approved dimensions when specified. Otherwise select representative modern dimensions and record them.

The score must include image-difference evidence, structural comparison, region-level scoring, responsive scoring, typography comparison, critical-component review, and a weighted total.

A weighted 99% does not pass if a critical region materially fails. Critical failures include missing CTA, missing navigation, unreadable text, broken layout, wrong section order, missing approved imagery, severe responsive drift, obstructed controls, or clipped critical content.

If objective scoring is unavailable, report:

`VISUAL PARITY: NOT VERIFIED`

Never round a failing score upward.

## 10. Recursive repair loop

After each preview deployment:

1. run BrowserWorker validation
2. score visual parity
3. run operational acceptance tests
4. classify failures
5. identify root causes
6. generate repair work packets
7. dispatch branch-safe repairs
8. verify changes
9. redeploy preview
10. revalidate with BrowserWorker

Run up to five repair cycles unless all gates pass, the operator stops the run, a protected action requires approval, failures repeat without progress, a connector becomes unavailable, or the repair requires secrets, payment, spend, destructive action, or production mutation.

Each cycle records attempt number, commit SHA, deployment ID, routes, visual score, operational score, failures, dispatched changes, tests, screenshots, console errors, network errors, result, and remaining blockers.

After five unsuccessful cycles, block release, preserve the latest working preview, list exact differences and failures, and provide the next repair. Never declare false success.
