## Purpose

Resolve the dependency-audit blocker caused by the vulnerable `workflow` package without redesigning the canonical AUTO BUILDER frontend/control-plane.

## Inspection Summary

The previous matrix canary proved:

- `workflow@3` is not available from the registry evidence.
- `workflow@2.0.6` installs but does not expose `workflow/api`.
- `workflow@latest` resolves to `workflow@4.3.1`, preserves `workflow/api`, but remains dependency-audit blocked.

The source inspection for this pass verified:

- `workflow/api` was used by:
  - `src/app/api/cron/recursive-control/route.ts`
  - `src/app/api/workflows/awos-recursive-control/route.ts`
- `workflow/next` was used by:
  - `next.config.js`
  - `next.config.mjs`
- `.well-known/workflow` routes are generated by the workflow package during build and are not source-owned route files in `src/app`.
- The repo already has an internal recursive runtime function in `workflows/awos-recursive-control.ts` that materializes queues, telemetry, approval holds, sandbox checks, and runtime evidence.

## Repo Changes

- Replaced `workflow/api` startup with direct internal calls to `awosRecursiveControlWorkflow` in:
  - `src/app/api/cron/recursive-control/route.ts`
  - `src/app/api/workflows/awos-recursive-control/route.ts`
- Removed `workflow/next` wrappers from:
  - `next.config.js`
  - `next.config.mjs`
- Removed the direct `workflow` dependency from `package.json`.
- Replaced the package-version canary with an internal-runtime canary in:
  - `.github/workflows/workflow-dependency-canary.yml`

## Behavioral Decision

The branch now uses an internal direct runtime for the recursive control loop instead of the external `workflow` package runtime.

This intentionally removes dependency on package-generated `.well-known/workflow` routes. The replacement keeps the governed queue/telemetry/control-plane behavior in repo-owned code and removes the vulnerable dependency cluster.
